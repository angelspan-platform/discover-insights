Preparation Everything that happens before a prompt is run. If this is skipped or done poorly, results from the experiment are unreliable.
Before any stage Setup
Do
  • Read SPL transcripts and extract discrete statements into a spreadsheet — one statement per row, include speaker role
  • Write the workshop prompt for each community (one sentence framing the problem)
  • Set up Claude Projects with system prompt and uploaded files
  • Prepare a blank evaluation log with one row per MVP behavior
Consider
  • Is each extracted statement standalone, or does it only make sense in context?
  • Is the workshop prompt specific enough, or so broad any theme could be justified against it?
  • Are speaker roles being captured accurately for gap alert testing?
Produce
  • Inputs spreadsheet (75–100 inputs minimum)
  • Configured Claude Projects, ready to run
  • Blank evaluation log
Do
  • Read all MVP behaviors in the framework
  • Understand the clustering prompt and the brief generation prompt before running either
  • Confirm the batch-of-5 process and what to record after each run
Consider
  • What does a good theme suggestion look like vs. a vague or premature one?
  • What am I specifically watching for during each batch?
Produce
  • Clear understanding of each prompt and what to log
Do
  • Review the SPL engagement context and recall what themes came out of the real engagement
  • Define personal approval standards before the experiment starts — write them down
Consider
  • What themes would I expect the AI to surface?
  • What is the bar for approval — specificity, accuracy, recognizability to contributors?
Produce
  • Written approval standard, defined before any AI output is seen
Do
  • Set up the evaluation log with one row per MVP behavior
  • Read and understand the pass/fail criteria for each behavior in the framework
Consider
  • What specific evidence would constitute a pass for each behavior?
  • What would a clear failure look like?
Produce
  • Evaluation log template, populated and ready to record observations
Stage 2 Theme Identification — Feed inputs in batches of 5. Observe how the AI clusters and names themes. Moderator reviews and approves.
Theme Identification MVP
Do
  • Hand inputs spreadsheet to the Prompter
  • Be available to answer questions about extraction decisions
Consider
  • Were there ambiguous inputs during extraction that the Prompter should know about?
Produce
  • Clarifications on specific inputs as needed (supporting role)
Do
  • Paste first 5 inputs and run the clustering prompt
  • Record the AI response, then add 5 more and run again
  • Repeat until all inputs are processed
  • Log theme suggestions after every batch — do not filter or summarize
Consider
  • Is the AI holding back with little data, or suggesting themes prematurely?
  • Are theme suggestions changing significantly between batches, or stabilizing?
  • Are labels specific ("Invoice-to-Cash Handoff") or generic ("Process Issues")?
  • Deliberately run at least one batch of conflicting or sparse inputs
Produce
  • Running log of all theme suggestions by batch number
  • Note of the batch at which suggestions first appeared and when they stabilized
Do
  • Review all theme suggestions at end of the full run
  • For each suggestion: decide Approve, Rename, or Reject
  • Write one sentence of rationale for every decision
  • Flag any inputs that were not assigned to any theme
Consider
  • Would contributors recognize their input in this theme?
  • Is the label accurate, or has the AI subtly reframed what was said?
  • Are any two themes actually the same problem described differently?
  • Are unassigned inputs important signals the AI missed?
Produce
  • Approved theme list with written rationale for every decision
Do
  • Record which batch produced the first theme suggestion
  • Track when suggestions stabilized across batches
  • Log all moderator decisions (Approve / Rename / Reject) and rationale
  • Note failure modes: premature clustering, generic labels, missed inputs
Consider
  • Is the AI's behavior consistent with the framework pass criteria?
  • What specifically passed or failed, and why?
  • What would need to change in the prompt to address failures?
Produce
  • Completed Stage 2 rows in the evaluation log
Stage 3 Discussion Deepening — Test whether AI can generate specific follow-up questions and identify missing voices. Requires approved themes from Stage 2.
Discussion Deepening V1
Not active at this stage
Do
  • Take individual inputs from each approved theme
  • Run the follow-up prompt against a minimum of 10 inputs per theme
  • Record every generated question verbatim — do not paraphrase
Consider
  • Is the AI generating one question per input, or more than one?
  • Does the question reference something specific from the input, or is it generic?
Produce
  • Log of generated questions paired with their source inputs
Do
  • Review each generated question alongside its source input
  • Rate each question: Specific and useful / Generic / Misleading
Consider
  • If a real participant received this question, would they provide more useful detail than they already gave?
  • Could this question have been written without reading this specific input?
Produce
  • Rated question log with one-word assessment per question
Do
  • Review all question ratings and identify patterns
  • Note which input types produce better questions (long vs. short, factual vs. opinion)
  • Flag any questions that could apply to any input regardless of content
Consider
  • What is the ratio of specific to generic questions?
  • At what volume or type of input does quality degrade?
Produce
  • Completed Stage 3 rows in the evaluation log
  • Pattern summary: which conditions produce better or worse questions
Stage 4 Brief Synthesis — Generate a structured brief from approved theme inputs. The brief is the primary deliverable. Evaluated against the framework and against known SPL outcomes.
Brief Synthesis MVP
Not active at this stage
Do
  • Feed the full input set for each approved theme into the brief generation prompt — run once per theme
  • For SPL: also run a thin-data version (fewer inputs than ideal) to test whether the AI flags its own low confidence
Consider
  • Does the brief cover all required sections as defined in the framework?
  • Is the structure consistent across themes?
Produce
  • Full brief output per approved theme
  • Thin-data brief for SPL (comparison run)
Do
  • Read each brief as if presenting it to a client
  • Verify every quoted voice against source inputs — any unverifiable quote is an automatic failure
  • Assess overall integrity: does the brief represent the data or distort it?
Consider
  • Would I put my name on this and present it in a client meeting?
  • Would the client recognize their team's input in these findings?
  • Are the opportunity areas something that could be acted on, or are they restatements of the theme?
Produce
  • Written assessment per brief: Approve for client use / Requires rework / Would not use
  • One sentence of reasoning for each assessment
Do
  • Compare brief themes against Stage 2 approved themes — they must match
  • Check each opportunity area for specificity against the pass criteria
  • For SPL: compare brief output against known engagement outcomes
  • Score each section of every brief against the framework pass criteria
Consider
  • Is the brief technically accurate but too generic to be useful?
  • Are tensions grounded in specific inputs, or asserted without evidence?
  • Does the thin-data version flag its own uncertainty, or overstate confidence?
Produce
  • Completed Stage 4 rows in the evaluation log
  • Overall pass/fail summary across all MVP behaviors
Throughout Documentation — Ongoing responsibility across all stages. The log of observations is the primary output of the experiment, not the AI's responses.
All stages Ongoing
Active only in Preparation
Do
  • Record all AI responses verbatim — never summarize or paraphrase
  • Note exactly which prompt and inputs produced each response
  • Log unexpected results immediately, before moving to the next batch
Consider
  • An unexpected result is as valuable as a pass — record it with the same care
Produce
  • Complete verbatim log of AI responses by stage and batch
Do
  • Write rationale for every decision at the time it is made — not retrospectively
  • Do not revise decisions after seeing subsequent AI outputs
Consider
  • Moderator decisions made in the moment are the data — they must reflect genuine judgment, not hindsight
Produce
  • Contemporaneous decision log with rationale per decision
Do
  • Update framework status after each stage completes — do not batch updates
  • Write one sentence per behavior observed: what happened, not just pass or fail
Consider
  • The evaluation log is what drives prompt refinement — vague entries produce vague improvements
Produce
  • Fully populated evaluation log after each stage
  • Final experiment summary used to refine prompts before the next community run
